## Project Overview

AnaOS is a SaaS platform for managing credit cooperatives ("cooperativas"). It provides multi-tenant management of cooperativas, agencies (agencias), accounts (cuentas), users, and subscription plans. The system is deployed on Railway.

**Target market:** Cooperativas of any size (flexible SaaS plans: básica / professional / enterprise).
**Core value proposition:** Analytics and reporting dashboard that gives cooperative managers real-time visibility into their operations.

## Repository Structure

## Commands

```bash

dotnet build                      # Build

```

**Local dev with Docker (dotnet not installed locally):**
```bash
# Start postgres
docker run -d --name anaos-postgres -e POSTGRES_USER=postgres -e POSTGRES_PASSWORD=admin \
  -e POSTGRES_DB=AnaOSDB -p 5434:5432 postgres:16-alpine

# Build and run backend
docker build -t anaos-backend .
docker run -d --name anaos-backend \
  -e "ConnectionStrings__DefaultConnection=Host=host.docker.internal;Port=5434;Database=AnaOSDB;Username=postgres;Password=admin" \
  -e JWT_KEY="dev-secret-key-1234567890-abcdefgh" \
  -e PORT=8080 -p 8080:8080 anaos-backend
```

> Important: `DATABASE_URL` env var always appends `SSL Mode=Require`, which breaks local postgres.
> For local dev, use `ConnectionStrings__DefaultConnection` directly to bypass SSL.

## Backend Architecture

### Layer Pattern
Controllers → Services → Repositories → EF Core DbContext → PostgreSQL

Each domain entity has a corresponding interface + implementation at every layer:
- `Interfaces/I<Entity>Repository.cs` / `Interfaces/I<Entity>Service.cs`
- `Repositories/<Entity>Repository.cs` (extends `BaseRepository<T>`)
- `Services/<Entity>Service.cs` (extends `BaseService<T>`)
- `Controllers/<Entity>Controller.cs`

`BaseRepository<T>` and `BaseService<T>` provide generic CRUD. Domain-specific repositories/services override or extend these.

### Domain Entities
- **Cooperativa** — tenant root; each cooperativa owns its agencies, accounts, and subscription
- **Agencia** — branch of a cooperativa; a "Matriz" agency is created automatically on signup
- **Usuario** — users with role-based access; Administrador/Gerente have no `IdCooperativa`, all others must have one (enforced by DB check constraint)
- **Cuenta** — financial accounts belonging to a cooperativa
- **PlanSuscripcion** — subscription plans (basica / professional / enterprise / custom)
- **Suscripcion** — active subscription linking a cooperativa to a plan
- **HistorialSuscripcion** — audit log of subscription state changes
- **Socio** — *(planned)* cooperative member/client (not yet implemented)

### Authentication & Authorization
- Passwords hashed with SHA256 (`Custom/Utilidades.cs`) using lowercase hex (`x2` format)
- JWT tokens generated by `Utilidades.GenerarJWT()`, expire in 60 minutes
- Token claims include: `idUsuario`, `email`, `role`, and optionally `idCooperativa`
- `RoleAuthorizationMiddleware` extracts claims into `HttpContext.Items` (`UserId`, `UserRole`, `IdCooperativa`)
- `[RequireRole("Administrador", "Gerente")]` custom attribute (`Attributes/RequireRoleAttribute.cs`) enforces role access at the action/controller level
- `AccesoController` is `[AllowAnonymous]`; all other controllers require `[Authorize]` + `[RequireRole]`

### User Roles
| Role | Access |
|---|---|
| `Administrador` | Global platform admin; no cooperativa |
| `Gerente` | AnaOS manager; no cooperativa |
| `Cooperativa` | Cooperative owner/manager |
| `UsuarioCooperativa` | Regular user within a cooperative |

### Database
- PostgreSQL via Npgsql + EF Core
- Schema is created using `EnsureCreated()` (not migrations) — avoid `dotnet ef migrations` unless intentionally switching
- `AppContext.SetSwitch("Npgsql.EnableLegacyTimestampBehavior", true)` is set globally — store all `DateTime` values as `DateTimeKind.Unspecified`
- `OnModelCreating` in `AnaOSDbContext` defines all table names, keys, relationships, and check constraints

### Environment Variables (Backend)
| Variable | Purpose |
|---|---|
| `DATABASE_URL` | PostgreSQL URL `postgresql://user:pass@host:port/db` — Railway production only (forces SSL) |
| `ConnectionStrings__DefaultConnection` | Direct Npgsql connection string — use for local dev (no SSL) |
| `JWT_KEY` | JWT signing key (overrides `appsettings.json`) |
| `PORT` | HTTP port; defaults to `8080` |
| `ALLOWED_ORIGINS` | Comma-separated list of allowed CORS origins appended to defaults |

Local development uses `appsettings.Development.json`; `appsettings.json` has placeholders that must be overridden via env vars in production.

### API Response Convention
All endpoints return a consistent shape:
```json
{ "isSuccess": true/false, "message": "...", "data": ... }
```

## Frontend Architecture

### Key Libraries
- **React Router v6** — client-side routing with `ProtectedRoute`
- **TanStack Query** — server state management
- **shadcn/ui** + **Radix UI** — component library
- **Tailwind CSS** — styling
- **React Hook Form** + **Zod** — form validation
- **Recharts** — charts/analytics
- **Sonner** — toast notifications
- **Kushki** — payment gateway integration (`KushkiPaymentForm` component)

### State & Auth
- `AuthContext` (`src/contexts/AuthContext.tsx`) holds the authenticated user; wraps the whole app
- `apiService` (`src/services/api.ts`) is the single HTTP client; stores JWT and user in `localStorage`
- `ProtectedRoute` component checks authentication and `allowedRoles` before rendering pages
- `AppContext` (`src/contexts/AppContext.tsx`) holds application-level state separate from auth

### Routing
Routes are defined in `App.tsx`. Most routes are wrapped in `<ProtectedRoute>`. Admin-only routes use `allowedRoles={['Administrador']}`.

### API Connection
The frontend connects to the backend using `VITE_API_BASE_URL` (build-time env var, e.g. `https://api.railway.app/api`). Falls back to `http://localhost:5133/api` for local dev.

## Deployment (Railway)

Both services are deployed separately on Railway.

- **Backend**: Docker multi-stage build (SDK 9.0 → ASP.NET 9.0 runtime). Health check: `GET /api/PlanSuscripcion`.
- **Frontend**: Docker multi-stage build (Node 20 → Nginx alpine). `VITE_API_BASE_URL` must be set as a Railway build variable. `nginx.conf` uses `envsubst` to inject the dynamic `PORT`.

### Signup Flow
`POST /api/Acceso/RegistroCompleto` atomically creates: Cooperativa → Agencia "Matriz" → Usuario with role `Cooperativa`. Files (logo, nombramiento) are sent as base64 strings and stored as `byte[]` in the DB.
